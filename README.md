## 総括
今回の課題を通して、アプリケーションのパフォーマンスチューニングにおいて、ボトルネックを正確に特定し、根拠を持って改善することの重要性を学びました。

まず、make_posts関数のN+1問題を解消した時、コード上ではクエリの呼び出し回数は減ったもののベンチマークのスコアは伸びませんでした。このことから、アプリケーションのパフォーマンスを考える際はコードロジックだけでなく、データベース側についてもセットで考える必要があることを実感しました。複合インデックスの追加によって実際にスコアが劇的に上がったのは嬉しかったです。

また、もう一つ行った変更としてdigest関数のリファクタリングがあります。このリファクタリングでは外部コマンドを呼び出すオーバーヘッドの大きさをベンチマークのスコアを通じて実感しました。コードの変更量でいえばたった1行ですが、スコアが約1000点増加したのは言語標準の機能を最大限使うことの重要性を感じました。

技術面以外でも、今回はプルリクエストの作成ルールについて詳細な指示をいただき、実務を意識した開発フローを体験できました。変更の粒度を適切に保つことや、レビュワーに対して「何のために、どのような変更を行ったか」を明確に伝えるDescriptionの記述は、チーム開発において不可欠なスキルであると深く理解できました。
